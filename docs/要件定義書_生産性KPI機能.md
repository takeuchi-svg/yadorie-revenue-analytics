## 1. 背景と目的

### 背景
YADORIE GROUPは生産性KPI管理表（OKJ_2025生産性KPI管理表）で、
施設別の労働生産性を管理している。現在はExcelで手集計しており、
売上・PL・労働時間を別々に管理して手動で突き合わせている。

### 目的
既存BIに労働時間データ（勤怠CSV）を統合し、生産性KPIを自動算出・可視化する。
これにより、施設横断の生産性比較、月次推移の把握、人件費効率の分析が
リアルタイムで可能になる。さらに蓄積される日次勤怠データは、
将来のシフト管理・AIシフト予測の基盤データとなる。

### ゴール
- 生産性KPI管理表の「生産性関連」セクションをBI上で完全再現
- 勤怠CSVをアップロードするだけで生産性KPIが自動算出される
- 施設別・月別・施設間比較で生産性を分析できる

## 3. 機能要件

### 3.1 勤怠CSV取込

**入力**: KING OF TIME系の日次勤怠CSV（拡張子.xls、実体HTML、UTF-8）
- 1ファイル = 1日分・全従業員（約845行）
- 月末に1日ずつ出力 → 1ヶ月分（約30ファイル）をまとめてアップロード

**処理**:
- HTMLテーブルをパース
- 所属コード抽出 → 施設マッピング（dim_facility_mapping）
- 雇用区分判定（正社員=月給 / アルバイト=時給）
- ヘルプ勤務判定 → 労働時間を計上先施設に按分
- 打刻時刻の正規表現抽出（FeliCa/編/C/位置 等の形式に対応）
- 労働時間を分単位に正規化
- raw_attendance_daily にUPSERT（同日同施設は上書き）
- dim_staff を自動更新（新規社員番号を追加）

**エラー処理**:
- 本社部門（HQ）のレコードも取り込むが、施設別KPIからは除外
- 打刻なし（欠勤・休日）の行は労働時間0で記録
- 同じ日のCSVを再アップロードしても重複しない

### 3.2 生産性KPI算出

データベース設計書の mart_productivity ビューで以下を自動算出:

| KPI | 計算式 | データソース |
|-----|--------|------------|
| 売上高人件費率 | 人件費 ÷ 売上 | PL ÷ 売上 |
| 社員数（月給） | DISTINCT(正社員) | 勤怠 |
| アルバイト数（時給） | DISTINCT(アルバイト) | 勤怠 |
| 総労働時間 | SUM(労働合計) | 勤怠 |
| 総残業時間 | SUM(残業系) | 勤怠 |
| 月給社員1人あたり平均残業時間 | 総残業 ÷ 社員数 | 勤怠 |
| 従業員1人1時間あたりの売上 | 売上 ÷ 総労働時間 | 売上 ÷ 勤怠 |
| 従業員1人1時間あたりの付加価値 | GOP ÷ 総労働時間 | PL ÷ 勤怠 |
| 1部屋あたりの労働時間 | 総労働時間 ÷ 販売室数 | 勤怠 ÷ 売上 |
| 顧客1人あたりの労働時間 | 総労働時間 ÷ 客数 | 勤怠 ÷ 売上 |
| みなし残業超の残業代 | （手動入力） | 手動 |
| 労働時間（派遣・その他） | （手動入力） | 手動 |

### 3.3 手動入力（設定画面）

Settings に「生産性手動入力」タブを追加:
- 施設選択 + 月選択
- みなし残業超の残業代（円）
- 派遣・その他の労働時間（時間）
- 保存 → dim_productivity_manual にupsert

### 3.4 生産性ページ（UI）

サイドバーの「ANALYSIS」グループに「Productivity（生産性）」を追加。

**画面構成**:
1. ヘッダー: 施設セレクタ + 月セレクタ + データソースバッジ（勤怠+PL+売上）
2. KPIカード上段（4枚）: 売上高人件費率 / 総労働時間 / 1人1時間あたり売上 / 1人1時間あたり付加価値
3. KPIカード下段（4枚）: 社員数 / アルバイト数 / 総残業時間 / 1部屋あたり労働時間
4. 月次推移チャート: 1人1時間あたり売上（bar + line、前年比較）
5. 施設間比較: 全施設の生産性ランキング（横棒、当月）
6. 労働時間内訳: 所定/残業/深夜/休日の積み上げ棒
7. AIコメンタリー（表示のみ）

各KPIカードにはYoY比較（前年同月比）を表示。

## 4. 非機能要件

- **パフォーマンス**: 27施設×12ヶ月の生産性KPIを1秒以内に表示（ビューのインデックス最適化）
- **データ整合性**: 勤怠の総労働時間と、生産性KPI管理表の手集計値が一致すること（検証必須）
- **UI統一**: 既存BIのダークテーマ・配色・コンポーネントに準拠（UI仕様書参照）
- **施設分離**: 全施設のデータが同一テーブルに入るが、facility列で完全分離

## 5. データフロー

```
[月次運用]
  勤怠システム（KING OF TIME）
    → 日次CSV出力（1日ずつ、月末に30日分）
    → Upload画面「勤怠取込」にまとめてドロップ
    → HTMLパース → raw_attendance_daily にUPSERT
    → dim_staff 自動更新

  予実管理表（既存）
    → raw_pl_actual（実装済み）

  売上（既存）
    → raw_reservation（実装済み）

  手動入力
    → Settings「生産性手動入力」→ dim_productivity_manual

[表示]
  mart_labor_monthly（労働時間集計ビュー）
  mart_productivity（生産性KPIビュー）
    ← raw_attendance_daily + raw_pl_actual + raw_reservation + dim_productivity_manual をJOIN
    → 生産性ページに表示
```

## 7. 受け入れ基準

- [ ] 山の手（FRY）の勤怠CSV（1ヶ月分）をアップロードして総労働時間が表示される
- [ ] 総労働時間が生産性KPI管理表の手集計値と一致する
- [ ] 1人1時間あたり売上 = 売上÷総労働時間 が正しく算出される
- [ ] ヘルプ勤務の労働時間が計上先施設に正しく按分される
- [ ] 本社部門の労働時間が施設KPIに混入しない
- [ ] 手動入力（みなし残業超残業代、派遣時間）が反映される
- [ ] 施設間比較で全施設の生産性が横棒で表示される
- [ ] 月次推移で前年比較が表示される
- [ ] 既存の売上分析・PL予実の表示に影響がない

---

## 8. リスクと対策

| リスク | 対策 |
|--------|------|
| 勤怠CSVのHTML形式が施設/月で微妙に異なる | パーサーをアンカー方式（カラム名ベース）で実装 |
| ヘルプ勤務の按分ミス | テストデータで按分結果を検証 |
| raw_pl_actualのカラム名が想定と違う | 実装前にClaude Codeで既存テーブル定義を確認 |
| 30ファイル一括アップロードの処理時間 | プログレスバー + バッチ処理 |
| 打刻形式の多様性（FeliCa/編/C/位置） | 正規表現を網羅的にテスト |
