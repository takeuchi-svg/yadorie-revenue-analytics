# 運用手順書

日常運用（月次の取込・随時のマスタ更新・DDL適用・アカウント管理）の手順。

## 1. 月次ルーティン（毎月やること）

### 1-1. 売上データ取込（各施設）
アップロード画面 `/upload` → 「売上データ」タブ。デフォルト施設を選んでから、その月のCSVをまとめてドロップ。
- 対象: PMS(Staysee) 5種（予約/基本商品/その他商品/入金/販売数集計表）＋ サイトコントローラ(Lincoln) ＋ レート表。
- **ファイル名に対象年月（例 `2026_05`）を含めること**。明細系（基本/その他商品/入金/販売数/レート）は年月が読めないと**取込を拒否**する（再取込での重複倍加を防ぐため）。
- 取込後、結果に「⚠ n行スキップ」が出たら、ヘッダー名の変化やフォーマット差異を確認する（黙って欠落しない設計）。

### 1-2. PL実績取込（各施設）
`/upload` →「PL実績」タブ → 会計の損益計算書CSV。**同一施設×年度を上書き**。

### 1-3. 勤怠取込（全施設一括）
`/upload` →「勤怠」タブ → Touch On Time の日次出力（.xls/HTML）を1ヶ月分（約30ファイル）まとめて。
- 全施設一括のため施設選択は不要。ヘルプ按分・施設マッピングは自動。
- **「⚠ 未マッピング所属 n行スキップ」が出たら新施設/新部署の可能性** → `dim_facility_mapping` と `src/lib/etl/attendance-parser.ts` の `FACILITY_MAP`/`NAME_FAC` に追加が必要。
- 権限: 全施設分を含むため **admin が実施**（member は自施設外の行が拒否される）。

### 1-4. スポット実働（居れば）
シフト画面 `/shift` → スポット行に実働時間を入力 → 保存（実績 `raw_attendance_daily(source='manual')` に記録され生産性KPIへ反映）。

> 生産性の「みなし残業超の残業代」「派遣・その他労働時間」は**手入力不要**（勤怠×賃金から自動算出＝T13）。

## 2. 随時（変化があったときだけ）

- **賃金設定**: 新入社員（勤怠取込で自動的に従業員一覧へ）・昇給・時給改定・スポット追加時に、設定 `/settings`「従業員 賃金設定」で入力。毎月は不要。**給与閲覧権限**が必要（adminは常に可）。
- **稼働日数**: 予実の在庫数算出用。設定「稼働日数（月別）」。
- **OTAマーケ費用**: 設定「OTAマーケティング費用」。
- **施設マスタ（総客室数）・勤務パターン/役割**: 設定（**編集はadminのみ**）。
- **年度予算**: 計画スプレッドシートから取込（budget-import スキル）。

## 3. DDL適用（スキーマ変更）

**直DB接続は現環境で不通のため、Supabase SQL Editor で手動実行する**（`scripts/migrate.mjs` のpg直結は使わない。使う場合は環境変数 `SUPABASE_DB_URL` が必要）。

1. Supabase Dashboard → SQL Editor → New query
2. `scripts/sql/<対象>.sql` の全文を貼り付け → Run（RLS警告は「Run and enable RLS」）
3. 冪等（何度実行しても安全）に作ってある。**ただし下記「適用順」と「⚠再実行注意」を厳守**。

### 3.1 全体を1から張り直す場合の適用順（正本）

同名のビュー・関数を後発ファイルが `create or replace` するため、**順序を誤ると最新の修正が古い定義で上書きされる**。ゼロから構築する場合は必ず以下の順で流す（棚卸2026-07-21で確定）:

1. **基盤**: `setup.sql`
2. **予約日ベース分析**: `booking_analysis_setup.sql` → `onhand_setup.sql` → `lincoln_retire.sql` → `lincoln_drop.sql`（リンカーン=raw_booking_event を廃止しステイシー/raw_reservation一本化）
3. **シフト・労務**: `shift_setup.sql` → `shift_seed.sql` → `shift_pattern_settings.sql` → `shift_pattern_fry_scope.sql` → `shift_extras.sql` → `shift_variance_v2.sql` → `capex_staffing.sql`
4. **予算・予実**: `budget_version.sql` → `budget_lock.sql` → `budget_review.sql` → `company_opening_date.sql` → `company_opening_date_seed.sql`
5. **稼働・集計の是正**: `calc_audit_fixes.sql` → `mart_occupancy_calendar.sql` → `facility_type.sql` → `facility_profile.sql`
6. **AIナレッジ基盤**: `mart_ai_setup.sql` → `mart_ai_fix.sql` → `ai_knowledge_setup.sql` → `ai_knowledge_k20*.sql` → `ai_knowledge_k40.sql` → 各 `*_prompt*.sql` / `*_insight*.sql` / `meeting_*.sql` / `review_*.sql` / `company_insight.sql` / `booking_insight.sql`
7. **人件費モデルv2**: `labor_model_v2.sql` → `labor_model_v2_phase3.sql`（`dim_staff_wage` を撤去し `dim_labor_rate`/`raw_regular_labor_monthly` へ）
8. **権限（必ずAI基盤より後）**: `rls_facility.sql` → `wage_permission.sql`
9. **並び順・軽微**: `staff_sort_order.sql` / `facility_sort_order.sql`
10. **RLS是正**: `rls_hardening_2026_07.sql`（`dim_labor_rate`/`raw_regular_labor_monthly` を施設スコープへ・施設コードFK）
11. **最後に必ず**: `security_invoker_fix.sql`（ビューの security_invoker とゲートを最終確定）

> 🔎 **RLS健全性チェック**: 上記適用後（および新テーブル追加時）に `rls_audit.sql` を Run。**0行＝健全**。行が返ったら「RLS未有効／全開ポリシー（using true）／施設ゲート不在」のいずれかが残っている。加えて anon（未ログイン）で全オブジェクト0件を Node の2クライアント走査で確認する。

### 3.2 ⚠ 単体で再実行すると壊れるファイル（後勝ち注意）

現在のライブDBは正しい状態。**以下を単体で流し直すと退行**するため、触る場合は §3.1 の順で末尾まで通すか、`security_invoker_fix.sql` を必ずセットで最後に流すこと:

- `rls_facility.sql` … `is_admin()` を再定義。owner対応は `ai_knowledge_setup.sql` と一致させ済（両方 role in ('admin','owner')）だが、**古い版を混ぜないこと**。
- `labor_model_v2_phase3.sql` / `shift_variance_v2.sql` … `mart_labor_cost_monthly` / `mart_shift_variance_monthly` を再作成（`invoker=off`＋`can_access_facility` ゲート内蔵済。ゲート無しの旧版に戻さないこと）。
- `mart_occupancy_calendar.sql` / `budget_version.sql` / `shift_extras.sql` … 各ビューに `invoker=on` を明示済。
- `mart_ai_setup.sql` … 冒頭で `drop schema mart_ai cascade`。**再実行したら必ず `mart_ai_fix.sql`・`budget_version.sql`・`mart_occupancy_calendar.sql` を後続で流し直す**（さもないと灯がアクセス権限エラーで無応答に戻る）。
- `calc_audit_fixes.sql` … `raw_booking_event`（リンカーン・撤去済）に依存する旧6ビュー定義を含む。**lincoln_drop 後は単体再実行不可**（存在しないテーブル参照で失敗）。cxl/booking_lt/gs/plan/adr_band 系は現行の raw_reservation ベース定義（lincoln_retire.sql）が正。
- `labor_model_v2.sql` … `dim_labor_rate`/`raw_regular_labor_monthly` のRLSは施設スコープ（`can_access_facility`）に修正済。**旧 `using(true)`（全宿丸見え）に戻さないこと**。`shift_setup.sql` は `dim_role`/`dim_shift_pattern` 等に `using(true)` を張るが、これらはマスタ（全員読みで可）。ただし書込みも全開なので、マスタ編集はアプリUI側（設定画面）でガードしている前提。

### 3.3 権限モデルの前提（member追加時に必須）

- `admin` / `owner` = 全施設アクセス（`is_admin()` で通過。`user_facility` 登録は不要）。
- `member` = **`user_facility` に割当行がある施設だけ**閲覧可。**割当が無いと全施設で何も表示されない**（設定「ユーザー管理」で必ず閲覧施設を選ぶこと）。

## 4. アカウント管理

設定「ユーザー管理」（admin のみ）。
- 発行: メール＋初期パスワード＋役割（一般/管理者）。一般は閲覧施設を選択。
- **給与閲覧**: 一般ユーザーの行のチェックで、割当施設の従業員の賃金・個人別人件費の閲覧/編集を許可。
- 一般ユーザーは割当施設のデータのみ（DBのRLSで強制）。給与閲覧OFFなら賃金は非表示・シフトの人件費タイルは「-」。

## 5. デプロイ

main へ push で Vercel が自動デプロイ（takeuchi-svg / beta.vercel.app）。環境変数 `ANTHROPIC_API_KEY`・`SUPABASE_SERVICE_ROLE_KEY` 等は Vercel 側に設定。

## 6. トラブル時

- ページに**赤「データ取得エラー」** = 通信/権限の問題。再読込。→ 黄「データ未登録」（＝未取込）とは別物なので、再アップロードしない。
- 一般ユーザーで何も表示されない = 施設割当が未設定。ユーザー管理で確認。
- DDL実行で権限エラー = 自分がadmin登録済みか `select role from app_user where user_id=auth.uid()` で確認。
