# データベース全体設計（正本）

実在する全テーブル/ビューの台帳と、システム横断の前提。**スキーマに関する疑問はまずここ。** 個別機能の背景は各設計書、実行するDDLは `scripts/sql/*.sql`。

- DB: Supabase(PostgreSQL)。命名規約 `raw_`(生データ) / `dim_`(マスタ) / `mart_`(集計VIEW)。例外テーブル（budget_*/actual_monthly/ai_*/app_user/user_facility）は歴史的経緯で無接頭辞。
- 施設キー: `facility` / `work_facility` / `home_facility` = **TEXT のBIコード**（FRY, NS…）。サロゲートID無し。全結合の軸。
- 時間軸: 月=TEXT `'YYYY-MM'`、日付=DATE、勤怠/シフトの労働時間=**分(整数)**。
- **mart_ は全て通常VIEW**（MATERIALIZED VIEWは無い）＝毎クエリで raw を集計する。→ 70施設スケール時はマテリアライズ化を検討（§既知の負債）。

## 2. 予算の3系統（混同注意）

1. **dim_budget**（レガシー・月別）: `mart_monthly_kpi` の revenue_budget/total_inventory が参照。未メンテだと概要のocc/revpar分母がNULLになる。
2. **budget_monthly**（月次PL）: 予実管理(yojitsu)・`mart_budget_revenue_monthly` が参照。予実の正。
3. **budget_daily**（日別）: 日別売上(daily)・シフトの稼働前提が参照。日付列は `date`。

→ 将来 dim_budget を budget_monthly へ一本化するのが望ましい（§既知の負債）。

## 3. 施設コードとマッピング

- BIコード（FRY等）が全テーブルの施設キー。27施設→2030年70施設へM&A拡大予定。**施設追加は原則データ追加のみ**（`dim_facility`＋`dim_facility_mapping`に行追加）。
- **例外**: 勤怠パーサー `src/lib/etl/attendance-parser.ts` の `FACILITY_MAP`/`NAME_FAC`（TS定数）に施設を足す必要がある＝コード変更＋デプロイ。70施設化前にDB駆動化したい負債。
- 施設コードは2系統: 勤怠所属コード（数値"121"）↔ BIコード("FRY")。変換は `dim_facility_mapping`。
- HQ（本社部門）は施設KPIから除外（`work_facility <> 'HQ'`）。

## 4. システム横断の前提（AIエージェント/実装者向け・◆=誤解しやすい）

- ◆ 稼働率・販売室数の正は **raw_room_sales / mart_occupancy_***。`raw_reservation.room_count` は全件1固定で**使用禁止**。`mart_monthly_kpi.rooms_sold` は SUM(nights) で室数と厳密には異なる（ADRが過大寄り）。
- ◆ 売上は2系統: raw_reservation の精算額（status='C/O', nights>0）と PLの `sales_total`。人件費率はPL同士、生産性は raw_reservation/mart を使い分け。
- ◆ PL(actual_monthly/budget_monthly)は freee由来。**「人件費」カテゴリも「GOP」行も無い**。全費目が `category="売上高"`。人件費=item_name合計（給料手当+賞与+通勤費+法定福利費+福利厚生費+雑給+外注費(人材)、実績は外注費が合算の場合あり）。付加価値≒`gross_profit`（売上−原価）。
- ◆ `fiscal_year` は4月開始（'2026'=2026/4〜2027/3）。売上の月判定は checkin ベース `TO_CHAR(checkin,'YYYY-MM')`。
- ◆ 勤怠のヘルプ勤務は**計上先施設**に計上。時間は分。労働時間の小数時間は×60で分に正規化（Touch On Time）。
- ◆ 人件費: 時給=実働×時給／月給=月給+超過×(月給/所定×1.25)。**割増1.25のみ**。月給baseは**本務施設のみ**計上、残業判定は従業員×月の全施設合計時間（クロス施設二重計上を防止）。
- ◆ スポット判定は `dim_staff.is_spot`（attendanceに列は持たない）。スポット実働は raw_attendance_daily(source='manual')。
- localStorage禁止（施設選択はCookie）。

## 5. 権限モデル（RLS）

- 関数 `is_admin()` / `can_access_facility(f)` / `can_view_wage()`（`scripts/sql/rls_facility.sql`・`wage_permission.sql`）。
- **admin**=全施設・全操作。**member**=`user_facility` の施設のみ読み書き。マスタ(dim_facility等)=全員read・admin書き。
- **賃金(dim_staff_wage)・個人別人件費ビュー**=給与閲覧権限(`app_user.can_view_wage`)者のみ。施設×月合計(mart_labor_cost_monthly)は施設メンバー可。
- mart_ ビューは `security_invoker=on` で閲覧者のRLSを継承（例外: mart_labor_cost_monthly は invoker=off＋ビュー内ゲート）。
- AI API(/api/chat,/api/insight)はBearer認証＋memberは許可施設フィルタをサーバー強制（service_roleがRLSをバイパスするため）。

## 7. 既知の負債（優先度順・レビュー由来）
