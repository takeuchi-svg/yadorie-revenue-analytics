## 1. アーキテクチャ（本番）

```
① CSVダウンロード（Staysee 5本 + Lincoln 2本 + レート表 1本）
    ↓
② /upload 画面にドラッグ&ドロップ（ログイン必須）
    ↓
③ ブラウザ内ETL: ファイル自動判定 → cp932デコード → パース → Raw層オブジェクト生成
    ↓
④ Supabaseに書込（ブラウザの supabase クライアントから直接 upsert / delete+insert）
    ↓
⑤ ダッシュボード: 各画面ロード時に mart_* ビューを SELECT → 動的集計を表示
```

- フレームワーク: Next.js 16 (App Router) + TypeScript + Tailwind CSS
- DB / 認証: Supabase (PostgreSQL) + Supabase Auth (email/password)
- チャート: Recharts（仕様書のチャート種別・配色に従う）
- CSV: PapaParse + TextDecoder(cp932) / Excel: SheetJS(xlsx)
- マイグレーション: `scripts/migrate.mjs`（pg 直接接続。service_role ではなく DB パスワードで接続）

## 3. Raw層スキーマ（実装済み DDL の要点）

§ raw_reservation / raw_basic_product / raw_other_product / raw_payment / raw_booking_event /
raw_rate_snapshot は `scripts/migrate.mjs` の DDL がSSOT。主キー/一意制約は以下。

- `raw_reservation`: `UNIQUE(facility, pms_id)` — upsert
- `raw_booking_event`: `UNIQUE(facility, notify_no)` — upsert（Lincoln重複排除）
- `raw_rate_snapshot`: `UNIQUE(facility, snapshot_date, stay_date, scope, COALESCE(room,''))` — delete+insert
- `raw_basic_product` / `raw_other_product` / `raw_payment`: 一意キーなし → **施設+source_month で delete+insert**（重複防止）

### 3.1 raw_room_sales（販売数集計表 / 確定販売室数）★本番追加

**粒度**: 1行 = 宿泊日 × スコープ（客室タイプ別 or 合計）
**ソース**: Staysee「販売数集計表」CSV（ピボット表：行=客室タイプ＋合計、列=日1..31、値=その日の販売室数）
**重複防止**: 施設+source_month で delete+insert

```sql
CREATE TABLE raw_room_sales (
  id BIGSERIAL PRIMARY KEY,
  facility TEXT NOT NULL,
  stay_date DATE NOT NULL,
  scope TEXT NOT NULL,            -- 'type'（客室タイプ別） / 'total'（合計）
  room_type TEXT,                -- scope='total' のとき NULL
  sold INTEGER NOT NULL DEFAULT 0,
  source_month TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE UNIQUE INDEX idx_room_sales_unique
  ON raw_room_sales(facility, stay_date, scope, COALESCE(room_type, ''));
CREATE INDEX idx_room_sales_facility_date ON raw_room_sales(facility, stay_date);
```

検出キーワード: ファイル名「販売数集計表」。変換: 日列を縦展開し、合計行を scope='total'、各タイプ行を scope='type' で投入。

## 4. Mart層（稼働率）★重要

### 4.1 稼働率の正データは raw_room_sales

```sql
CREATE OR REPLACE VIEW mart_occupancy_daily AS
SELECT rs.facility, rs.stay_date AS date, TO_CHAR(rs.stay_date,'Dy') AS dow,
       rs.sold AS rooms_sold, f.total_rooms,
       CASE WHEN f.total_rooms > 0 THEN ROUND(rs.sold::NUMERIC / f.total_rooms, 4) END AS occ
FROM raw_room_sales rs
LEFT JOIN dim_facility f ON rs.facility = f.facility
WHERE rs.scope = 'total';

CREATE OR REPLACE VIEW mart_occupancy_monthly AS
SELECT rs.facility, rs.source_month AS month,
       SUM(rs.sold) AS rooms_sold, COUNT(DISTINCT rs.stay_date) AS operating_days,
       f.total_rooms,
       CASE WHEN f.total_rooms > 0 AND COUNT(DISTINCT rs.stay_date) > 0
         THEN ROUND(SUM(rs.sold)::NUMERIC / (f.total_rooms * COUNT(DISTINCT rs.stay_date)), 4) END AS occ
FROM raw_room_sales rs
LEFT JOIN dim_facility f ON rs.facility = f.facility
WHERE rs.scope = 'total'
GROUP BY rs.facility, rs.source_month, f.total_rooms;
```

検証: 山の手4月 = 202室、稼働率67.3%（202 / (10室×30日)）。

### 4.2 既知の課題：予約ベースの稼働率は不正確

`mart_monthly_kpi.occ` / `mart_daily.occ` は `raw_reservation.nights` から算出しているが、
**これは不正確**。理由：

- Staysee「予約情報」CSVには室数カラムが無い。
- 「部屋名」はカンマ区切りで複数室が入り、客室＋ダイニング＋宴会場が混在する。
- そのため `raw_reservation.room_count` は全件 `1` 固定になっており、信用できない。

→ **稼働率・販売室数は必ず `raw_room_sales` / `mart_occupancy_*` を使用する**こと。
→ ダッシュボードのOCC表示は `mart_occupancy_*` に接続する（移行作業中／完了後はこちらが正）。

## 7. 重複防止ルール（運用上の要点）

| テーブル | 方式 | キー |
|---------|------|------|
| raw_reservation | upsert | facility, pms_id |
| raw_booking_event | upsert | facility, notify_no |
| raw_rate_snapshot | delete+insert | facility, snapshot_date |
| raw_basic_product / other_product / payment | delete+insert | facility, source_month |
| raw_room_sales | delete+insert | facility, source_month |

→ 同じ月・同じファイルを何度アップロードしても二重にならない。施設ごとに完全分離。
