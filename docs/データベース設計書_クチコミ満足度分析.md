# クチコミ・満足度分析 データベース設計書（ドラフトv1）

**対象**: YADORIE売上分析BI への クチコミ(WEB) + 館内アンケート 統合分析機能の追加
**先行施設**: 山の手ホテル(FRY)
**前提**: 既存Supabase（raw_/mart_/dim_ 3層構造）に統合。同一DB・別テーブル方式。
**スコープ外**: クチコミへの返信管理・AI返信案生成（reply_textカラムのみ将来用に確保）

---

## 1. 設計方針

1. **WEBクチコミと館内アンケートは別テーブル**（粒度・項目が異なるため混ぜない）
2. **評価軸マッピングで統合**: じゃらん「風呂」・楽天「お風呂」・アンケート「風呂」を
   同一axis_codeに正規化し、mart層で横比較可能にする
3. **原文は原文のまま保存**（Raw層の原則）。AI分析結果は別テーブルに分離し、
   分析ロジック改善時に再実行・上書き可能にする
4. **統計補正はmart層**: ベイズ平滑化（事前分布=施設の長期平均24ヶ月、k=10）
   + 直近3ヶ月ローリングを主表示に
5. **全テーブルにfacility + 日付軸**: 既存の売上・PL・生産性とJOIN可能
   （AIエージェントの横断分析が最終ゴール）

## 2. テーブル定義

### 2.1 fact_review（WEBクチコミ）

1行 = 1クチコミ。

```sql
CREATE TABLE fact_review (
  id BIGSERIAL PRIMARY KEY,
  facility TEXT NOT NULL REFERENCES dim_facility(facility),
  source TEXT NOT NULL,              -- 'jalan' | 'rakuten' | 'ikyu' | 'google'
  source_review_id TEXT,             -- ソース側ID（じゃらん=クチコミ管理番号、一休=予約ID等）
  booking_no TEXT,                   -- ★予約番号（raw_reservation.booking_noとJOIN可能）
  review_date DATE NOT NULL,         -- 投稿日
  stay_date DATE,                    -- チェックイン日（じゃらん・一休CSVは正確に取得可）
  overall_rating NUMERIC(3,1),       -- 総合点（ソースのスケールのまま保存）
  rating_scale INTEGER NOT NULL DEFAULT 5,
  sub_ratings JSONB,                 -- ソース別項目別評価（キーはソース原文のまま）
  title TEXT,
  body TEXT,                         -- 原文そのまま
  reviewer_attr JSONB,               -- {"gender":"男性","age_group":"50代","scene":"一人旅","plan":"...","room":"...","price":41250,"post_site":"一休"} 取れる範囲
  reply_text TEXT,                   -- 施設返信（今回スコープ外、カラムのみ確保）
  ingested_via TEXT,                 -- 'api' | 'csv' | 'manual'
  created_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE (facility, source, source_review_id)
);
CREATE INDEX idx_fact_review_fac_date ON fact_review(facility, review_date);
CREATE INDEX idx_fact_review_booking ON fact_review(booking_no);
```

**sub_ratingsのキーはソース原文のまま入れる**。正規化は 2.4 のマッピングで行う。

**★booking_no による予約データ結合（本設計の要）**:
じゃらんCSVの「予約番号」・一休CSVの「予約ID」は、PMS/リンカーンの予約番号と同一体系
（検証済み: PMS↔リンカーンで99.7%一致）。これにより
`fact_review JOIN raw_reservation ON booking_no` で、クチコミと
プラン・客室・チャネル・精算額・宿泊日を紐づけた分析が可能
（例:「低評価が集中するプラン」「単価帯×評価の相関」）。
実装時にKT/FRYの実データでマッチ率を検証すること。

### 2.1b fact_review_summary（楽天用: 集計値のみのソース）

楽天は管理画面から個別クチコミをダウンロードできない（確認済み）。
宿カルテ「口コミ分析」の集計値を月1回手動転記する。

```sql
CREATE TABLE fact_review_summary (
  id BIGSERIAL PRIMARY KEY,
  facility TEXT NOT NULL REFERENCES dim_facility(facility),
  source TEXT NOT NULL DEFAULT 'rakuten',
  month TEXT NOT NULL,               -- 'YYYY-MM'
  review_count INTEGER,              -- 当月投稿件数
  overall_avg NUMERIC(3,2),          -- 総合評価（累積表示値）
  axis_scores JSONB,                 -- {"サービス":4.5,"立地":4.2,"部屋":4.4,"設備・アメニティ":4.1,"風呂":4.3,"食事":4.6}
  area_ranking TEXT,                 -- '54位/104施設' など参考値
  created_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE (facility, source, month)
);
```

楽天の個別クチコミ本文は任意（余力時に定型テンプレで手動投入→fact_reviewへ）。
トピック分析の母数はじゃらん・一休・Google・アンケートで確保する。

### 2.2 fact_survey_response（館内アンケート）

1行 = 1回答。Googleフォーム→（GAS or CSV）→本テーブル。

```sql
CREATE TABLE fact_survey_response (
  id BIGSERIAL PRIMARY KEY,
  facility TEXT NOT NULL REFERENCES dim_facility(facility),
  response_at TIMESTAMPTZ NOT NULL,  -- 回答日時
  stay_date DATE,                    -- 宿泊日（フォームで選択 or 滞在中回答なら当日）
  survey_version TEXT NOT NULL,      -- 'v1' 設問構成のバージョン
  expectation_gap SMALLINT,          -- Q1: 4=はるかに超えた/3=超えた/2=どおり/1=届かず
  score_clean SMALLINT,              -- Q2: 清潔感 1-5
  score_service SMALLINT,            -- Q3: 接客 1-5
  score_dinner SMALLINT,             -- Q4: 夕食 1-5
  score_breakfast SMALLINT,          -- Q5: 朝食 1-5
  score_bath SMALLINT,               -- Q6: 風呂 1-5
  low_score_reason TEXT,             -- 分岐: 低評価時の理由（自由記述）
  relax_moment SMALLINT,             -- Q7: くつろぎ実感 4段階
  most_memorable TEXT,               -- Q8: 選択値 'food'|'bath'|'room'|'service'|'scenery'|'other'
  most_memorable_other TEXT,         -- Q8: その他の自由記述
  good_point TEXT,                   -- Q9: 良かったこと（自由記述）
  improvement_point TEXT,            -- Q10: もっと良かったのに（自由記述）★改善示唆の主源泉
  nps SMALLINT,                      -- Q11: 0-10
  raw_answers JSONB,                 -- フォーム回答の生JSON（将来の設問変更に備え全保存）
  created_at TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX idx_survey_fac_date ON fact_survey_response(facility, response_at);
```

**設計判断**: YADORIE標準設問(v1)は正規化カラムで持ち、同時にraw_answersに生JSONも保存。
設問改訂(v2)時は survey_version で区別し、共通カラムはそのまま比較可能。

### 2.4 dim_axis_mapping（評価軸 統一マッピング）

WEBクチコミのsub_ratingsキー / アンケート設問 → 統一軸コードへの変換表。

```sql
CREATE TABLE dim_axis_mapping (
  source TEXT NOT NULL,              -- 'jalan'|'rakuten'|'ikyu'|'google'|'survey'
  source_key TEXT NOT NULL,          -- 'bath' / 'お風呂' / 'score_bath' など
  axis_code TEXT NOT NULL,           -- 'bath'（統一コード）
  PRIMARY KEY (source, source_key)
);
```

**統一軸コード（実CSVのキーに基づく確定版）**:
| axis_code | 名称 | じゃらん(CSV実キー) | 一休(CSV実キー) | 楽天(宿カルテ) | Google | アンケート |
|---|---|---|---|---|---|---|
| overall | 総合 | 総合評価 | レイティング(個人総合) | 総合評価 | rating | -（期待ギャップは別扱い） |
| room | 部屋 | 部屋 | レイティング(客室・アメニティ) | 部屋 | - | -（v1未設問） |
| bath | 風呂 | 風呂 | レイティング(温泉・お風呂) | 風呂 | - | score_bath |
| dinner | 夕食 | 料理夕食 | - | - | - | score_dinner |
| breakfast | 朝食 | 料理朝食 | - | - | - | score_breakfast |
| meal | 食事(総合) | -（夕朝の平均で導出可） | レイティング(食事) | 食事 | - | -（夕朝の平均で導出可） |
| service | 接客 | 接客・サービス | レイティング(接客・サービス) | サービス | - | score_service |
| clean | 清潔感 | 清潔感 | - | - | - | score_clean |
| facility_equip | 設備 | - | レイティング(施設・設備) | 設備・アメニティ | - | - |
| location | 立地 | - | - | 立地 | - | - |
| relax | くつろぎ実感 | - | - | - | - | relax_moment（YADORIE独自） |
| local | ローカル体験 | - | - | - | - | most_memorable（分布型） |

食事軸の注意: じゃらんは夕/朝が分離、一休・楽天は食事1本。mart層では
dinner/breakfastを持つソースから meal=AVG(dinner,breakfast) を導出し、
meal軸で全ソース比較可能にする（逆方向の分解はしない）。

### 2.5 fact_feedback_topic（AI分析結果: トピック×センチメント）

クチコミbody / アンケート自由記述をAIが分類した結果。**元データと分離**。

```sql
CREATE TABLE fact_feedback_topic (
  id BIGSERIAL PRIMARY KEY,
  facility TEXT NOT NULL,
  source_table TEXT NOT NULL,        -- 'fact_review' | 'fact_survey_response'
  source_id BIGINT NOT NULL,         -- 元レコードID
  source_field TEXT,                 -- 'body'|'good_point'|'improvement_point'|'low_score_reason'
  topic_code TEXT NOT NULL,          -- 'bath_temp','breakfast_variety','checkin_wait',...
  topic_label TEXT,                  -- 表示名「風呂の温度」
  sentiment TEXT NOT NULL,           -- 'positive'|'negative'|'neutral'
  quote TEXT,                        -- 該当箇所の抜粋
  model_version TEXT,                -- 分析モデルのバージョン（再実行管理）
  analyzed_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE (source_table, source_id, source_field, topic_code, model_version)
);
CREATE INDEX idx_topic_fac ON fact_feedback_topic(facility, topic_code, sentiment);
```

**トピック辞書は固定リストにせず、AIの自由抽出 + topic_codeへの名寄せ**をmodel_versionごとに管理。
将来pgvector導入時は fact_review / fact_survey_response 側に embedding カラムを追加（本設計では未導入）。

### 2.6 mart_guest_feedback（統合集計ビュー）

施設×月×軸×ソース区分の集計。**ベイズ平滑化 + 3ヶ月ローリング**。
楽天（fact_review_summary）は既に集計値のため、本ビューとは別に
UNIONで表示層に合流させる（平滑化対象外・件数バッジ付き表示）。

```sql
CREATE OR REPLACE VIEW mart_guest_feedback AS
WITH review_axis AS (
  -- WEBクチコミのsub_ratingsを統一軸に展開（5点満点に正規化）
  SELECT f.facility,
         to_char(f.review_date,'YYYY-MM') AS month,
         'web' AS channel,
         m.axis_code,
         (kv.value)::numeric * 5.0 / f.rating_scale AS score
  FROM fact_review f,
       LATERAL jsonb_each_text(f.sub_ratings) AS kv(key,value)
  JOIN dim_axis_mapping m ON m.source = f.source AND m.source_key = kv.key
  WHERE kv.value ~ '^[0-9.]+$'
  UNION ALL
  SELECT facility, to_char(review_date,'YYYY-MM'), 'web', 'overall',
         overall_rating * 5.0 / rating_scale
  FROM fact_review WHERE overall_rating IS NOT NULL
),
survey_axis AS (
  SELECT facility, to_char(response_at,'YYYY-MM') AS month, 'survey' AS channel,
         a.axis_code, a.score::numeric AS score
  FROM fact_survey_response s
  CROSS JOIN LATERAL (VALUES
    ('clean', s.score_clean), ('service', s.score_service),
    ('dinner', s.score_dinner), ('breakfast', s.score_breakfast),
    ('bath', s.score_bath),
    ('relax', (s.relax_moment * 5.0 / 4)::numeric)   -- 4段階→5点換算
  ) AS a(axis_code, score)
  WHERE a.score IS NOT NULL
),
unioned AS (
  SELECT * FROM review_axis UNION ALL SELECT * FROM survey_axis
),
monthly AS (
  SELECT facility, month, channel, axis_code,
         COUNT(*) AS n, AVG(score) AS raw_avg
  FROM unioned GROUP BY 1,2,3,4
),
longterm AS (  -- 施設×軸の長期平均（直近24ヶ月）= ベイズ事前分布
  SELECT facility, axis_code, AVG(score) AS lt_avg, COUNT(*) AS lt_n
  FROM unioned
  WHERE month >= to_char(now() - interval '24 months','YYYY-MM')
  GROUP BY 1,2
)
SELECT m.facility, m.month, m.channel, m.axis_code, m.n, 
       ROUND(m.raw_avg,2) AS raw_avg,
       -- ベイズ平滑化: k=10, 事前=施設長期平均。長期n<12なら補正せずNULL（参考値扱い）
       CASE WHEN l.lt_n >= 12
            THEN ROUND((m.n * m.raw_avg + 10 * l.lt_avg) / (m.n + 10), 2)
            ELSE NULL END AS smoothed_avg,
       (m.n < 5) AS is_low_sample
FROM monthly m
LEFT JOIN longterm l USING (facility, axis_code);
```

**3ヶ月ローリング**は同構造の `mart_guest_feedback_3mo`（month範囲をwindowで束ねる）を別ビューで定義（実装時にClaude Codeが同パターンで作成）。

### 2.7 mart_improvement_topics（改善候補TOP集計）

```sql
CREATE OR REPLACE VIEW mart_improvement_topics AS
SELECT facility,
       to_char(COALESCE(r.review_date, s.response_at::date),'YYYY-MM') AS month,
       t.topic_code, t.topic_label,
       COUNT(*) FILTER (WHERE t.sentiment='negative') AS negative_mentions,
       COUNT(*) FILTER (WHERE t.sentiment='positive') AS positive_mentions,
       COUNT(DISTINCT t.source_table || ':' || t.source_id) AS sources
FROM fact_feedback_topic t
LEFT JOIN fact_review r ON t.source_table='fact_review' AND t.source_id=r.id
LEFT JOIN fact_survey_response s ON t.source_table='fact_survey_response' AND t.source_id=s.id
GROUP BY 1,2,3,4;
```

→ BI「今月の改善候補TOP3」= negative_mentions降順。アンケートとWEB両方で言及されるトピックが上位に来る。

### 2.8 NPS集計（mart_nps）

```sql
CREATE OR REPLACE VIEW mart_nps AS
SELECT facility, to_char(response_at,'YYYY-MM') AS month,
       COUNT(*) AS n,
       ROUND(100.0 * (COUNT(*) FILTER (WHERE nps>=9) - COUNT(*) FILTER (WHERE nps<=6))
             / COUNT(*), 1) AS nps_score
FROM fact_survey_response
WHERE nps IS NOT NULL
GROUP BY 1,2;
```

## 3. データ収集ルート（初期実装）

| ソース | ルート（確定） | 頻度 | パーサー仕様 |
|---|---|---|---|
| じゃらん | 管理画面CSV（お客様管理＞ランキング・予約分析DL＞クチコミ/アンケート結果DL） | 月1 | cp932 / **1行目タイトル行スキップ(skiprows=1)** / 本文に改行含む→クォート対応必須 / 22列 |
| 一休 | 管理画面CSV（予約情報＞クチコミダウンロード） | 月1 | cp932 / フラット19列 / クチコミ投稿サイト列あり（一休/Yahoo!トラベル） |
| 楽天 | **DL機能なし（確定）**→ 宿カルテ集計値を手動転記(fact_review_summary) + 個別本文は任意手動 | 月1 | 転記フォーム or 定型テンプレ |
| Google | R1: 手動/CSV → R2: **GBP API自動** | R2で日次 | マーケ担当にGBP管理者追加を依頼（1回） |
| 館内アンケート | Googleフォーム+QR → R1: CSVエクスポート → R2: GAS自動連携 | R1月1/R2日次 | フォーム回答Sheets→取込 |

**ファイル自動判定**: じゃらんCSV=ヘッダーに「クチコミ管理番号」を含む / 
一休CSV=ヘッダーに「レイティング(個人総合)」を含む、で判別（既存Upload画面の判定方式を踏襲）。

**初期投入**: 過去2年分（1施設 約100-400件）を手動/CSVで一括投入。遡及は一回きり。

**取込UI**: 既存Upload画面に「クチコミ取込」タブ追加。
- 定型CSVテンプレート（source,review_date,overall_rating,sub_ratings...）を配布し、手動分はこれに貼る
- UNIQUE制約（facility,source,source_review_id）でUPSERT重複排除
- source_review_idが無い手動分は `manual-{日付}-{連番}` を自動採番

## 4. アンケート運用フロー（山の手 R1）

```
Googleフォーム(v1・11問・条件分岐付き)
  → QRコード（客室カード + チェックアウト時案内）
  → 回答は自動でスプレッドシート蓄積
  → R1: 月次CSVエクスポート → Upload画面から取込
  → R2: GASで日次自動連携（Supabase REST API直POST）
```

フォームは施設ごとに1つ（facility埋め込み済みURL）。設問はdim_survey_question(v1)と同期。
